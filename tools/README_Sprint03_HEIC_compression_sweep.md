# Sprint 03: HEIC Compression Sweep

Mac-side compression sweep for the Nereus/Bristlemouth reef reference card workflow.

This script starts from the **high-quality spatial-density outputs** generated by Sprint 02 and tests how much HEIC compression can be applied before AprilTag/card analysis degrades.

Sprint 02 result used as the default resolution band:

```text
3072x1728: PASS
2688x1512: PASS
2304x1296: PASS
1920x1080: PASS
1600x900: PASS
1280x720: WARN
1024x576: WARN
854x480: WARN
640x360: FAIL
```

Therefore Sprint 03 defaults to this spatial ladder:

```text
3072x1728
2688x1512
2304x1296
1920x1080
1600x900
```

HEIC quality ladder:

```text
10 20 30 40 50 60 70 80 90
```

Lower HEIC quality means smaller file / more compression. Higher quality means larger file / less compression.

## Files

```text
bm_reference_card_heic_compression_sweep.py
bm_reference_card_quality_v2.py       # existing analyzer dependency, unchanged
```

The HEIC wrapper uses `bm_reference_card_quality_v2.py` as the metrics engine. It adds the Sprint-specific HEIC encode/decode loop, strict all-four-tag PASS/WARN/FAIL logic, and cellular message-count estimates.

## Dependencies

Python dependencies are the same as the earlier card analyzer:

```bash
python3 -m pip install opencv-contrib-python pillow numpy
```

HEIC encoding/decoding backend:

- Default on macOS: built-in `sips`
- Alternative: ImageMagick via `magick`

Check whether `sips` is available:

```bash
which sips
```

If `sips` fails for HEIC on your Mac, install ImageMagick and rerun with `--codec-backend magick`:

```bash
brew install imagemagick
```

## What the script does

For each selected crop mode and resolution, the script:

1. Finds the Sprint 02 high-quality JPEG source image.
2. Encodes that image to HEIC at quality 10, 20, 30, ... 90.
3. Measures the compressed `.heic` payload size.
4. Computes payload-only cellular message counts:

```text
messages_300b = ceil(heic_size_bytes / 300)
messages_900b = ceil(heic_size_bytes / 900)
```

5. Decodes the HEIC back to PNG for analysis.
6. Runs `bm_reference_card_quality_v2.py` on the decoded PNGs.
7. Exports unscaled 1:1 reference-card crops from the decoded HEIC PNGs for visual inspection.
8. Applies strict Sprint PASS/WARN/FAIL logic:

```text
PASS:
  all required AprilTags detected
  min tag side >= 18 px

WARN:
  all required AprilTags detected
  10 px <= min tag side < 18 px

FAIL:
  missing one or more required AprilTags
  or min tag side < 10 px
```

The default required tags are the current reference card corners:

```text
tl:0,tr:1,bl:2,br:3
```

---

## Important note on message counts

The message counts are **payload-only engineering estimates**:

```text
ceil(HEIC file bytes / chunk payload size)
```

They do **not** include:

- Bristlemouth framing
- Sofar/SBD/Notecard queue overhead
- message headers
- application metadata
- retries/retransmits
- base64 or transport encoding overhead, if added later

Use them for apples-to-apples comparison across HEIC quality settings, not as final billed/cellular transport accounting.

## Most important result files

### 1. `results/heic_threshold_summary.csv`

One row per crop mode and resolution.

Key columns:

```text
crop_mode
output_resolution
lowest_pass_heic_quality
first_warn_heic_quality
first_fail_heic_quality
recommended_min_heic_quality
recommended_heic_size_bytes
recommended_heic_size_kb
recommended_messages_300b
recommended_messages_900b
status_sequence_low_to_high_quality
notes
```

This answers:

> At each spatial resolution, what is the most aggressive HEIC setting that still preserves all-four-tag detection?

### 2. `results/heic_global_recommendation_summary.csv`

One compact row per crop mode. It picks the smallest recommended HEIC payload among the per-resolution recommended PASS settings.

Key columns:

```text
crop_mode
recommended_output_resolution_by_smallest_payload
recommended_heic_quality
heic_size_bytes
heic_size_kb
messages_300b
messages_900b
tag_side_px_min
same_resolution_ref_psnr_rgb
card_ref_psnr_rgb
```

This is the fastest file to inspect if you want the likely transmission setting.

### 3. `results/heic_compression_quality_all.csv`

Full raw merged table for every tested image.

Key columns:

```text
crop_mode
resolution
heic_quality
quality_status
heic_size_bytes
heic_size_kb
messages_300b
messages_900b
tag_count
tag_ids
tag_side_px_min
same_resolution_ref_psnr_rgb
same_resolution_ref_laplacian_corr
ref_psnr_rgb
ref_laplacian_corr
heic_path
decoded_analysis_path
card_crop_1to1_path
card_crop_1to1_width
card_crop_1to1_height
```

Metric meaning:

- `same_resolution_ref_psnr_rgb`: decoded HEIC vs the high-quality JPEG at the same resolution. This isolates HEIC compression damage.
- `ref_psnr_rgb`: analyzer card-rectified comparison against the 3072x1728 reference image.
- `messages_300b` and `messages_900b`: estimated number of payload chunks required to transmit the compressed HEIC file.

## Suggested decision process

1. Start with `heic_threshold_summary.csv`.
2. For each resolution, find the lowest HEIC quality that still passes.
3. If the lowest PASS is immediately above WARN/FAIL, prefer one quality step higher for margin.
4. Compare message counts for 300-byte and 900-byte chunks.
5. Inspect the normalized cut sheet for quick comparison.
6. Inspect the `test_heic_card_crop_1to1_<mode>_<resolution>.png` sheet for true visual quality without cut-sheet downsampling.
7. Use the selected resolution/quality as the candidate production payload for a real Spotter/Bristlemouth transmit test.

For the current MVP, a sensible first place to inspect is likely:

```text
fixed 1920x1080
fixed 1600x900
auto 1920x1080
auto 1600x900
```

Sprint 02 showed 1600x900 still passing, but 1920x1080 had margin before the WARN transition.
